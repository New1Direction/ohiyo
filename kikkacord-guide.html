Shipped today · end-to-end by default

Signal-grade encryption.
Zero key handling.

Tap the lock in any chat and it goes end-to-end — forward-secret, on every device, even for whole groups. The server only ever stores ciphertext, and your keys never leave your hands. No QR codes, no key files, nothing to manage.

🔐
Forward secrecy, by Signal
X3DH key agreement and the Double Ratchet — the same protocol behind Signal and WhatsApp. Every message gets its own key, so even a stolen device can't unlock your past chats.
New today
📱
Multi-device
Sign in on a second device and it just reads your encrypted messages — keys fan out to every device you own.
New
⏳
Disappearing messages
Set a timer — 30 seconds to a week. Messages self-destruct on every device and on the server.
New
👥
Encrypted group chats
Real group end-to-end encryption with Sender Keys — one ciphertext, every member decrypts it, and a per-member signature means nobody can forge anybody. The server stays completely blind.
New
🧠
Locked-RAM key vault
On desktop your private keys live in page-locked, non-swappable memory — never sitting in plaintext on disk like everyone else. Sealed at rest, and burned the instant you ask.
New
🕯️
Dead man's switch
Go quiet for too long and your data wipes itself — your messages, and optionally your keys. You choose the window; Kikkacord holds the line if you can't.
New
🔒 What the server stores — and what only you can read

Built on the maintained libsignal protocol — no hand-rolled crypto. Verified end-to-end across two- and three-client tests.

Under the hood — every part

A real stack, not a wrapper.

Kikkacord is a Rust backend and a React 19 front-end shipped as a tiny native app — deployed and running today at kikkacord.fly.dev. Here's the whole thing, piece by piece.

🦀
Rust server
An axum HTTP API over sqlx + SQLite. Domain-split modules for auth, messages, channels, servers, roles, polls, events and more.
axum 0.8sqlxSQLite15 migrations
⚡
Realtime gateway
A WebSocket gateway with scoped broadcasts (channel / server / user). Typed events for messages, reactions, presence, typing, voice and read receipts.
WebSocketbroadcastone-time tickets
🔐
Auth & security
JWT + argon2 password hashing, one-time WS tickets (no token in URLs), SSRF-guarded link previews, per-IP rate limits, and access checks on every channel action.
JWTargon2SSRF guardsrate limits
🎥
Voice & video
Peer-to-peer WebRTC mesh — the server only relays signaling. STUN for discovery, TURN for tricky networks, plus camera and screen-share tracks.
WebRTCSTUN/TURNscreen-share
⚛️
React client
React 19 + TypeScript, Tailwind v4 and Vite. Virtualised message list, optimistic sends, per-channel drafts, and a warm hand-built design system.
React 19TypeScriptTailwind v4Vite
🖥️
Native desktop
Packaged with Tauri v2 into a ~20 MB native app (3.7 MB installer) — system WebView, deep links (kikkacord://) and native notifications. No Electron.
Tauri v2deep linksnative notifs
🧩
Plugin sandbox
Untrusted plugins run in a Web Worker with the DOM, auth token and network primitives stripped away. The host forwards only sanitized events over a narrow RPC.
Web Workerno DOM/tokensanitized RPC
🗄️
Data layer
SQLite on a persistent Fly volume — messages, attachments (with dimensions), read cursors, roles, polls and events. Migrations run automatically on boot.
SQLiteFly volumeauto-migrate

Deploy
A multi-stage Docker build (Rust → debian-slim) on Fly.io, live at kikkacord.fly.dev with a health check and a single always-on machine for the gateway.
Fly.ioDocker/healthz
✅
Quality gate
ESLint with hook-correctness as a hard error, 18 Playwright end-to-end suites covering every flow, and a CI workflow so quality can't regress.

How a message travels

You type & hit enter — an optimistic bubble appears instantly and is saved to a local outbox.
POST /messages — content is validated and written to SQLite on the server.
Gateway broadcast — the message is fanned out to everyone who can see the channel.
Instant delivery — recipients see it live over the WebSocket, no refresh.
Read & Seen — opening the chat acks it; in a DM the sender's bubble flips to "Seen".
Lost connection? — the outbox auto-flushes the moment you reconnect — nothing dropped.
